Projeto Final —
Backend Java Spring Boot

Desenvolvimento de uma aplicação backend com CRUD completo e consultas SQL nativas sobre o banco de dados projetado pela equipe.

Professor: Luís Carlos Costa Fonseca Disciplina: Banco de Dados Formato: Equipe (3–4 integrantes) Entrega: Apresentação + Repositório Valor: 10,0 pontos
01 —

Visão Geral do Projeto

Cada equipe deverá desenvolver uma aplicação backend em Java com Spring Boot que sirva de camada de acesso ao banco de dados projetado durante a disciplina. O objetivo é colocar em prática os conceitos de modelagem relacional, SQL e integração de banco de dados com uma aplicação real, seguindo boas práticas de desenvolvimento de software.

📌 Premissa

O banco de dados já foi projetado pela equipe (modelo ER + DDL). O projeto final consiste em construir a aplicação que o consome — expondo suas entidades via API REST e implementando consultas analíticas relevantes ao domínio do sistema.

Java 17+ Spring Boot 3.x Spring Data JPA MySQL 8+ REST API Git + GitHub
02 —

Requisitos do Projeto

REQ 01 CRUD de pelo menos 5 tabelas Obrigatório ▼

A equipe deve implementar operações Create, Read, Update e Delete completas para no mínimo 5 tabelas do modelo de banco de dados projetado. Cada tabela deve ser exposta como um recurso REST independente.

Para cada tabela, a equipe deve implementar:

  • Entity — classe Java mapeada com JPA/Hibernate (@Entity, @Table, relacionamentos)
  • Repository — interface que estende JpaRepository ou CrudRepository
  • Service — camada de regras de negócio (validações, tratamento de exceções)
  • Controller — endpoints REST com os verbos HTTP corretos

Endpoints mínimos por recurso:

  • GET /recurso — listar todos (com paginação, se possível)
  • GET /recurso/{id} — buscar por ID
  • POST /recurso — criar novo registro
  • PUT /recurso/{id} — atualizar registro existente
  • DELETE /recurso/{id} — remover registro
✅ Boa prática

Use DTOs (Data Transfer Objects) para separar a representação da API do modelo de banco de dados. Evite expor diretamente as entidades JPA nas respostas da API.

REQ 02 Consultas SQL Nativas — Relatórios e Análises Obrigatório ▼

A equipe deve implementar pelo menos 6 consultas SQL nativas (usando @Query(nativeQuery = true) ou EntityManager), distribuídas entre os três níveis abaixo. As consultas devem ser relevantes ao domínio do sistema da equipe e inspiradas nos temas trabalhados nas listas de exercícios da disciplina.

Distribuição mínima das consultas:

Nível 1 — Básico (mín. 2)
JOINs entre tabelas, filtros com WHERE, ordenação e agrupamentos com GROUP BY / HAVING
Exemplo de tema
Relatório de registros com dados de tabelas relacionadas; contagem ou soma por categoria/grupo
Nível 2 — Intermediário (mín. 2)
Subconsultas correlacionadas, EXISTS / NOT EXISTS, CTEs (WITH), ou Views chamadas via aplicação
Exemplo de tema
Registros que nunca apareceram em uma associação; ranking dos itens mais frequentes em cada grupo
Nível 3 — Avançado (mín. 2)
Window Functions (RANK, ROW_NUMBER, LAG/LEAD), chamadas a Stored Procedures, ou consultas com múltiplos CTEs encadeados
Exemplo de tema
Ranking por partição (ex.: top 3 por categoria); evolução temporal com comparação de períodos; análise acumulada

Temas de referência (das listas de exercícios da disciplina):

  • Relatórios com JOINs múltiplos e tradução de códigos numéricos (CASE WHEN)
  • Entidades sem associação — usando NOT IN vs. NOT EXISTS
  • Item mais vendido / mais frequente por grupo (subconsulta correlacionada)
  • Ranking dentro de partições usando RANK() ou DENSE_RANK()
  • Análise de evolução temporal com LAG() / acumulados com SUM() OVER
  • Chamada de Stored Procedure via @Procedure ou EntityManager
🔗 Como implementar no Spring

Use @Query(value = "SELECT ...", nativeQuery = true) no Repository para consultas simples, ou injete o EntityManager no Service para consultas dinâmicas com parâmetros variáveis. Cada consulta deve ser exposta em um endpoint GET /recurso/relatorio/nome-do-relatorio.

REQ 03 Organização do Projeto e Boas Práticas Obrigatório ▼

O projeto deve seguir a estrutura padrão de um projeto Spring Boot organizado em camadas, com separação clara de responsabilidades.

Estrutura de pacotes esperada:

  • br.uema.bd.entity — classes JPA mapeando as tabelas
  • br.uema.bd.repository — interfaces de acesso a dados
  • br.uema.bd.service — lógica de negócio e validações
  • br.uema.bd.controller — endpoints REST
  • br.uema.bd.dto — objetos de transferência de dados (opcional, mas recomendado)
  • br.uema.bd.exception — tratamento de exceções customizadas

Configuração mínima esperada:

  • Arquivo application.properties ou application.yml com datasource configurado
  • Tratamento de erros com respostas HTTP adequadas (404, 400, 500)
  • Repositório público no GitHub com README.md descrevendo o projeto e instruções de execução
⚠️ Atenção

Não coloque credenciais de banco de dados no repositório. Use variáveis de ambiente ou o arquivo application-local.properties listado no .gitignore.

REQ 04 Diferenciais (pontuação extra) Opcional ▼

A implementação dos itens abaixo não é obrigatória, mas agrega pontos à nota final da equipe:

  • Paginação e ordenação nos endpoints de listagem usando Pageable do Spring Data
  • Validação de entrada com Bean Validation (@NotNull, @Size, @Email, etc.)
  • Stored Procedure chamada pela aplicação — usando @Procedure ou CallableStatement
  • Trigger demonstrado — o banco possui um trigger implementado e a equipe demonstra seu acionamento via chamada à API
  • Documentação Swagger / OpenAPI — via springdoc-openapi, com a interface acessível em /swagger-ui.html
  • Testes de integração — pelo menos um teste por endpoint usando @SpringBootTest ou MockMvc

05 —

Dicas Técnicas

🚀 Ponto de partida sugerido
  • Gere o projeto em start.spring.io com as dependências: Spring Web, Spring Data JPA, MySQL Driver
  • Configure o application.properties com spring.jpa.hibernate.ddl-auto=validate (o schema já existe) ou none
  • Use o @RestController + @RequestMapping para organizar os endpoints por recurso
  • Teste os endpoints com Postman, Insomnia ou a extensão REST Client do VS Code antes da apresentação
🗂️ Chamando SQL nativo no Spring
  • Via Repository: @Query(value = "SELECT * FROM tabela WHERE ...", nativeQuery = true)
  • Via EntityManager: em.createNativeQuery("SELECT ...").getResultList()
  • Stored Procedure: @Procedure(procedureName = "nome_da_procedure") no Repository, ou StoredProcedureQuery via EntityManager
  • Para retornar projeções de múltiplas tabelas, use uma interface de projeção ou List<Object[]> com mapeamento manual no Service
